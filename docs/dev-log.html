1. Project Overview

Brit English Academy is a Cambridge-certified English school in Tarragona, Spain. Before this project, operations ran across three disconnected tools: a static website for marketing, WhatsApp groups for student communication, and PDF emails for homework/exams. This platform unifies all three into a single, production-ready web application.

Dual-Zone Architecture

  • Zone 1 — Public Website: SEO-optimised marketing site at the root domain. Built for conversion (visitor → registered student). Routes: /, /levels, /about, /blog, /contact.
  • Zone 2 — Private Community: Full learning management system behind authentication. Routes under /(app)/: dashboard, lessons, homework, exams, community, messages, admin.

2. Architecture

Route Group Strategy

Next.js App Router route groups are used to share layouts without affecting URL structure:

  • app/(auth)/ — login, signup, onboarding. No shared layout.
  • app/(app)/ — all authenticated routes. Shared layout provides the sidebar and auth guard server-side (redirects unauthenticated users to /login).
  • app/ root — public pages share SiteHeader + SiteFooter individually (not via a shared layout) so each public page controls its own full-page structure.

Server vs. Client Components

PatternUsed ForReason
Server Components (default)All pages, data fetchingEnables export const metadata, direct Supabase calls, no client JS overhead
Client Components ("use client")SiteHeader, DashboardSidebar, forms, interactive exam, file uploadRequires useState, usePathname, or browser APIs
AnimateIn wrapperFramer Motion scroll reveals on public pagesChildren are rendered on server; only the motion wrapper is client
Server ActionsContact form, admin operations, homework actionsKeeps logic on the server, avoids API route boilerplate

Data Fetching Pattern

All authenticated pages fetch data directly in the server component using the Supabase server client (createClient() from lib/supabase/server.ts). This client reads the session from HTTP-only cookies set by the Supabase SSR helper, so no client-side hydration is needed for initial data.

Admin-level storage access Pages that need to generate signed URLs for private Supabase Storage files (e.g. homework downloads) use the service-role admin client (createAdminClient()) server-side only. The service role key is never sent to the browser.

3. Tech Choices & Trade-offs

Framework: Next.js 16 App Router

✓ Chosen
Next.js App Router

First-class SSR + SSG, metadata API, route groups, server actions. Native Vercel deployment. Single repo for both public site and authenticated app.

Alternative
Separate SPA + API

React SPA + Fastify. More complex CORS setup, no SSR for SEO, two repos to maintain.

Database: Supabase (PostgreSQL)

✓ Chosen
Supabase

Managed PostgreSQL + Auth + Storage + Realtime in one service. Row-Level Security enforced at the database layer. Free tier covers early-stage load.

Alternative
PlanetScale / Neon + Auth.js

More flexible but requires separate auth, storage, and realtime solutions. Higher integration cost.

Styling: Tailwind CSS v4 + shadcn/ui

Tailwind v4 uses a CSS-first configuration via @import "tailwindcss" in globals.css. Brand tokens (navy, crimson, gold) are defined as CSS custom properties and referenced throughout the app as var(--color-primary) etc. shadcn/ui components use the base-nova style, which is compatible with Tailwind v4.

Note: Tailwind v4 differs significantly from v3. The tailwind.config.ts file is minimal — most configuration lives in globals.css.

AI: Claude API (claude-sonnet-4-6)

Claude was chosen over OpenAI GPT-4 for this project because:

  • Superior instruction-following for structured JSON output (exam grading rubrics)
  • 200K context window — can process long homework submissions in one call
  • Anthropic's model family aligns with the owner's preference

Email: Resend

Resend was chosen over SendGrid/Mailgun for its clean developer experience and Next.js-first SDK. In development, the email server action falls back to console.log so no real emails are sent during local testing.

Rate Limiting: Upstash Redis

Upstash provides serverless Redis via HTTP — compatible with Vercel Edge functions and Node.js environments. Rate limiting is applied to auth endpoints in middleware.ts using the @upstash/ratelimit sliding window algorithm.

RLS Policies

Every table has Row-Level Security enabled. Key policy patterns:

  • Students read own data: auth.uid() = student_id on homework, exam submissions, etc.
  • Students read published content: is_published = true on lessons and exams, filtered to level <= profile.level at the application layer.
  • Admin access: Uses a current_user_role() security-definer function to avoid infinite recursion when checking the profiles table from within a profiles RLS policy.
  • Community posts: All authenticated users can read; only the author or admin can delete.
  • Messages: Participants can only read conversations they are a member of.
-- Security-definer function (avoids RLS recursion on profiles)
CREATE OR REPLACE FUNCTION current_user_role()
RETURNS text
LANGUAGE sql
SECURITY DEFINER
STABLE
AS $$
  SELECT role FROM profiles WHERE id = auth.uid()
$$;

5. AI Integration

Homework Pre-scan

When a student submits homework, the server action in app/actions/homework.ts calls Claude to generate a preliminary grammar/structure report before the teacher reviews. This gives students instant feedback and saves teacher time.

Implementation

// Prompt sent to Claude for homework pre-scan
const prompt = `
You are an expert English language teacher.
Analyse the following student homework submission and provide feedback.
Return a JSON object with this exact structure:
{
  "overall_score": number (0-100),
  "cefr_level": string (A1-C2),
  "grammar_errors": [{ "text": string, "correction": string, "explanation": string }],
  "vocabulary_feedback": string,
  "structure_feedback": string,
  "positive_points": [string],
  "improvement_areas": [string],
  "summary": string
}

Student submission:
${submissionText}
`

Response parsing

Claude's response is extracted from the text content and parsed with JSON.parse(). The parsed object is stored in the claude_feedback jsonb column on homework_submissions. If parsing fails, a fallback object with the raw text is stored.

Exam Grading

Interactive exam open-text answers and PDF exam essays are graded by Claude. The prompt includes CEFR descriptors to ensure band scores are calibrated correctly.

Interactive exam prompt

// Claude grades each open-text question individually
const prompt = `
You are a Cambridge English examiner.
Grade the following student answer to this question.
Return JSON: {
  "score": number (0-100),
  "band": string (A1-C2),
  "feedback": string,
  "corrections": [{ "original": string, "corrected": string, "explanation": string }]
}

Question: ${question.text}
Student answer: ${studentAnswer}
`

Teacher override

Teachers can override Claude's score at any time from the admin homework review panel. The teacher_override_score field on exam_submissions takes precedence over score in all UI displays.

6. Authentication

Authentication uses Supabase Auth with the @supabase/ssr package for cookie-based sessions compatible with Next.js server components.

Flow

  1. User signs up → Supabase sends confirmation email
  2. User clicks link → /auth/callback route exchanges code for session
  3. After session is set → redirects to /onboarding if no level selected, else /dashboard
  4. All /(app)/ routes check auth in the shared layout server component — unauthenticated requests are redirected to /login

Middleware

middleware.ts runs on every request matching protected routes. It refreshes the session cookie (required by Supabase SSR), applies Upstash Redis rate limiting on auth endpoints, and guards /admin/* routes (redirects non-admin users to /dashboard).

Google OAuth Enabled in Supabase → Authentication → Providers. The callback URL must be added to the Google Cloud Console OAuth application's authorised redirect URIs: https://[supabase-ref].supabase.co/auth/v1/callback

7. Realtime (Messaging)

Private messaging uses Supabase Realtime channels. Each conversation has a dedicated channel named conversation:{id}. Messages are inserted into the database and the channel broadcasts the new message to all connected participants.

// Subscribe to a conversation channel
const channel = supabase
  .channel(`conversation:${conversationId}`)
  .on(
    'postgres_changes',
    { event: 'INSERT', schema: 'public', table: 'messages', filter: `conversation_id=eq.${conversationId}` },
    (payload) => setMessages(prev => [...prev, payload.new as Message])
  )
  .subscribe()

9. Deployment Pipeline

  1. Supabase (production project)
    • Create production Supabase project (separate from dev)
    • Run supabase/schema.sql + supabase/migrations/001_messaging.sql in SQL Editor
    • Create Storage buckets: homework (private), lessons (private)
    • Enable Google OAuth provider, add production callback URL
    • Add production domain to Authentication → URL Configuration → Redirect URLs
  2. Vercel
    • Connect GitHub repo (private) to Vercel
    • Set all environment variables in Settings → Environment Variables
    • Set NEXT_PUBLIC_APP_URL to https://www.BritEnglishAcademy.com
    • Deploy — Vercel auto-detects Next.js, zero config needed
  3. Custom domain
    • Add domain in Vercel → Settings → Domains
    • Point DNS CNAME www → cname.vercel-dns.com
    • SSL certificate provisioned automatically
  4. Resend domain verification
    • Add BritEnglishAcademy.com as a sending domain in Resend
    • Add DNS TXT + MX records as instructed by Resend

11. Known Limitations

  • Blog is static: The /blog page uses hardcoded post data. A Supabase-backed CMS was specified but not implemented. Adding a blog_posts table and dynamic /blog/[slug] route is the natural next step.
  • No image generation in production: The Replicate AI image generation integration scaffold is in place but requires a funded Replicate account and is not wired to any live UI trigger.
  • Lottie animations are basic: A service worker stub exists and placeholder JSON files are in public/animations/. Full Lottie integration on lesson completion and level-up events is partial.
  • PDF viewer: The PDF exam mode renders the PDF URL in an <iframe>. A dedicated PDF.js viewer would provide better cross-browser consistency.
  • No video upload to Supabase Storage: Lessons currently accept YouTube embed URLs. Direct video uploads would require a large-file multipart upload flow.
  • Homework file content extraction: Claude pre-scans the homework submission description (text field) rather than the actual file content. Extracting text from PDF/DOCX uploads requires a server-side parsing library (e.g. pdf-parse, mammoth).
  • Streak tracker: The dashboard displays a streak stat but the streak is calculated from lesson_completions timestamps. There is no dedicated daily-login tracking table.
